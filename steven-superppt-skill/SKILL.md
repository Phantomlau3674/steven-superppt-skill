# Steven SuperPPT Skill

## Core Principle

Treat every slide as two layers:

1. **Image plate**: use `$imagegen` or sourced imagery for the fuzzy parts: atmosphere, metaphor, texture, object mood, background wireframe, editorial scene, abstract depth, or visual direction.
2. **Precision layer**: use HTML, SVG, Canvas, or native PPTX shapes for the exact parts: text, labels, numbers, charts, axes, icons, layout grids, alignments, callouts, tables, and page furniture.

Never burn critical text, data, labels, or business logic into an AI-generated background. The image may inspire and frame; code or PPTX objects must carry the meaning.

Also treat HTML as a tool, not the medium. A deck made in HTML must still think like a presentation; a PPTX generated by code must still think like a designed artifact.

## Default Output Contract

When the user does not specify otherwise:

- Build for 16:9.
- Create or update `outline.md` when the task needs content planning.
- Create or update `references/style-reference-analysis.md` when the user provides style references.
- Create or update `references/imagegen-prompt-pack.md` when more than one generated plate or visual asset is needed.
- Create or update `references/asset-manifest.json` when any generated or sourced image asset is selected for the deck.
- Produce an editable PPTX when PowerPoint handoff matters.
- Produce an HTML preview when visual fidelity, browser rendering, or screenshot QA matters.
- Keep final artifacts in a stable project folder, not a temp-only scratch path.
- Include source files that can regenerate the deck.

## Public Routing

- Use this skill for high-aesthetic deck planning, style adaptation, hybrid composition, and multi-asset image planning.
- If the user only needs a simple PPT/PDF/HTML conversion, use a conversion-specific tool or presentation skill instead.
- If the user explicitly supplies a brand/template skill, follow that skill for brand rules and use this skill only for content planning, reference analysis, or hybrid production gaps.
- If the user only needs prose, a memo, or article content without slide output, do not use this skill.
- Use `$imagegen` for generated raster assets when available; if it is unavailable, switch to prompt-handoff mode and continue with sourced or programmatic assets.

## Workflow

### 1. Detect The Work Mode

Pick one or combine them:

- **Content planning mode**: the user gives a topic, goal, rough notes, meeting need, research packet, or asks to "策划 PPT / 做一期 PPT / 帮我想每页". First produce a polished slide plan before visual production.
- **Style reference mode**: the user provides screenshots, images, PPT/PDF, HTML, website links, brand assets, or says "参考这个风格 / 按这个样式". First extract a visual system and adaptation rules.
- **Production build mode**: the content and style are already clear, or the user asks directly for PPTX/HTML/PDF/images. Build and verify the deck.

For detailed playbooks and output schemas, read `references/workflow-contract.md`. Use `references/content-and-style-scenarios.md` only when extra examples are needed.

### 1.5 Choose Visual Asset Density

At skill startup, choose a deck-level `visual_asset_density` before detailed content planning or production:

- `lean`: restrained, information-first decks. Generate only cover, major section, and truly necessary metaphor plates.
- `standard`: default polished decks. Generate cover variants, one plate per major section, and key-slide visuals where they improve comprehension or emotion.
- `image-rich`: keynote, launch, product-demo, brand-story, campaign, or cinematic decks. Plan a visual asset for every slide, with extra variants for cover, product, transition, and hero slides.
- `auto`: infer the density from the brief, then record the resolved value.

Ask one concise choice when the user has not specified density and the task is not obviously auto-resolvable:

```text
请选择视觉资产密度：lean / standard / image-rich / auto。默认 standard；发布会、keynote、产品演示会自动按 image-rich。
```

Do not block on the question if the user wants immediate execution. Default to `standard`, but auto-upshift to `image-rich` for launch event, keynote, product reveal, visual-impact, Apple-style, campaign, cinematic, or demo-showcase requests. Auto-downshift to `lean` for legal, compliance, board update, financial report, dense analysis, or mostly text/table decks.

### 2. Lock the Brief

Capture or infer:

- audience and decision context
- deck goal: persuade, explain, report, teach, sell, or align
- content state: topic only, rough notes, researched facts, finished copy, or existing deck
- required output: PPTX, HTML, PDF, images, or all of them
- editability requirement in PowerPoint
- visual asset density: `lean`, `standard`, `image-rich`, or `auto` with resolved value
- brand constraints, reference decks, style images, fonts, colors, logos
- delivery setting: projector, mobile screenshot, printed PDF, meeting screen, or social post

If details are missing and the task is action-oriented, make reasonable assumptions and proceed. Mention assumptions in the final response.

### 3. Create The Canonical Outline

When content planning mode is active, do not jump straight to decoration.

Create or update `outline.md`. Treat it as the single source of truth for content and production. Do not maintain a separate story skeleton that can drift from it.

Every planned slide must have:

- `slide_id`
- `role`
- `job`
- `polished_headline`
- `on_slide_copy`
- `speaker_intent`
- `evidence_status`
- `visual_job`
- `asset_need`
- `imagegen_need`

Polish every page until the deck has a complete narrative arc. Remove vague placeholders such as "介绍背景", "核心优势", or "总结" unless they are rewritten into concrete claims. If a slide has no job, evidence, or emotional purpose, merge it or remove it.

### 4. Run The Style Reference Pass

When style reference mode is active:

- create or update `references/style-reference-analysis.md` before production
- list every reference input and path, including user-provided images, screenshots, decks, PDFs, webpages, logos, product photos, or brand assets
- inspect all user-provided reference materials before choosing a palette or layout
- extract tokens: type posture, palette, spacing, radii, surfaces, image treatment, chart treatment, motion/transition feel, density, and recurring motifs
- write what to keep, adapt, reject, and why
- translate the reference into a deck-specific visual system instead of cloning blindly
- if the reference contains real brand/product assets, keep provenance clear and do not fake missing assets
- state exactly which extracted tokens will be applied to this deck

For local image references, inspect with `view_image` when needed. For PPT/PDF/HTML references, extract screenshots or thumbnails when possible before summarizing the style.

### 5. Run The Aesthetic Preflight

For any deck that should feel premium, executive-ready, high-end, "not AI-looking", or visually distinctive, do this before layout:

- choose the medium persona: presentation designer, information designer, editorial art director, product demo designer, or motion designer
- if the direction is not locked, propose exactly three materially different directions: `safe_professional`, `kinetic_immersive`, `poetic_distinctive`
- define the core asset protocol: real logo, product image, UI screenshot, source document, chart data, or proof pixels before decorative style
- write the anti-slop bar: what must not appear, such as generic purple gradients, fake dashboards, decorative chips, meaningless glow, stock-like silhouettes, or text burned into images
- identify one hero slide or hero frame that gets the most craft investment
- create a compact visual-system note: type scale, palette, grid, motif, image treatment, chart treatment, and proof/evidence treatment

When looking for stronger taste anchors, read:

- `references/github-visual-methods.md` for visual rules and production routes
- `references/aesthetic-case-skills.md` for Huashu, Anthropic design skills, and design-skill case patterns from GitHub

### 6. Choose the Production Route

Use the route that fits the handoff:

- **Native PPTX route**: when the deck must be editable. Prefer template master layouts and placeholders if a template exists. Use PptxGenJS or python-pptx for shapes, text, charts, tables, and images.
- **HTML/Canvas route**: when visual precision, web typography, complex layout, animation, or screenshot QA matters. Render 16:9 HTML at a fixed coordinate system, then export to image/PDF/PPTX as needed.
- **Hybrid route**: when the final deck must look premium and remain mostly editable. Use image plates as backgrounds, then overlay native PPTX text/shapes or DOM-to-PPTX converted elements.
- **Markdown route**: when content iteration is the bottleneck. Use Marp/Slidev-style authoring for structure, then rebuild editable PPTX elements rather than relying on flat screenshot exports.

### 7. Plan And Generate Image Assets

Use `$imagegen` for generated raster plates. If `$imagegen` has not been loaded in the current turn, read its `SKILL.md` before generating images. Follow its default mode: built-in `image_gen` first, CLI fallback only when the user explicitly asks.

Use image generation for backgrounds, objects, textures, or wireframes only after the content plan, style reference pass, and aesthetic preflight are stable.

Honor the chosen `visual_asset_density`:

- `lean`: cover plate 1-2 variants; section plates only when they clarify structure; no decorative per-slide images.
- `standard`: cover plate 2-4 variants; one section plate per major chapter; key-slide plates for conceptual, product, comparison, or transition slides; subtle programmatic backgrounds for dense data slides.
- `image-rich`: every slide gets an asset plan; cover, hero, product, and transition slides get 2-4 variants; data and chart slides get low-density texture, spatial scaffolding, or abstract structure with clear safe zones, never fake dashboards.
- `auto`: resolve to one of the above before writing the prompt pack, and state the reason.

If the current environment cannot generate images directly, use prompt-handoff mode:

- still write `references/imagegen-prompt-pack.md`
- still write or update `references/asset-manifest.json`
- add `references/imagegen-handoff.md` with copy-ready prompts, intended tools, aspect ratio, safe zones, negative constraints, and expected filenames
- mark manifest entries as `planned` or `waiting-for-user-generated-asset`
- build the deck with neutral placeholders, sourced assets, or programmatic backgrounds when useful
- tell the user exactly which generated images they need to create and where to place them

Do not assume one generated image is enough. Decide the asset plan from `outline.md`:

- hero plate: usually 2-4 variants before selecting one
- section plates: one per major chapter if the deck needs chapter contrast
- per-slide visual metaphors: generate only when the slide needs material presence
- product/object plates: generate or source only when real assets are not required
- textures or wireframes: generate reusable background families, not random one-offs

For multi-asset generation, write `references/imagegen-prompt-pack.md` first. Each asset entry must include slide id, visual job, prompt, negative constraints, safe zones, overlay plan, variant count, expected output paths, and acceptance criteria.

For any generated or sourced image selected for the deck, update `references/asset-manifest.json` with source path, variants, selected path, final workspace path, selection reason, and usage status. For project-bound plates, move or copy selected final images from `$CODEX_HOME/generated_images/...` into the deck workspace, usually `assets/plates/` or `assets/generated/`. Never reference a project asset only from the generated-images cache.

Prompt for:

- aspect ratio and exact subject
- no text, no logos, no labels unless supplied as assets
- safe zones for title/body content
- visual metaphor, material, lighting, camera, and composition
- background contrast low enough for overlays
- room for charts or diagrams if needed

Reject image plates that compete with the message. A beautiful background that steals the hierarchy is a failed slide.

### 8. Compose the Precision Layer

Use a deterministic coordinate system:

- PPTX: 13.333 x 7.5 inches for 16:9.
- HTML preview: 1920 x 1080 or 1600 x 900 with fixed aspect-ratio scaling.
- Grid: use 8px/0.1in rhythm unless the brand template already defines another grid.

Prefer native text and shapes for:

- titles and subtitles
- data labels and chart values
- arrows, connectors, boxes, annotations
- legends, footnotes, page numbers
- tables and comparison matrices

Use Canvas/SVG for:

- dense diagrams
- custom chart geometry
- precise masks and paths
- generated line systems or particle-like visuals

### 9. Apply Visual Method Rules

When the visual direction is weak or the deck needs a premium pass, read `references/github-visual-methods.md`.

Always enforce:

- scan-first hierarchy: the viewer should understand the slide's main point within 3 seconds
- CRAP: contrast, repetition, alignment, proximity
- restrained type scale: usually 3-4 sizes and 2 weights
- semantic color: color must mean something, not only decorate
- sufficient contrast for projector and screenshot viewing
- generous negative space around the focal message
- repeated layout grammar with enough variation to avoid boredom

### 10. Verify

Do not call the deck done until it has been inspected as a visual artifact.

For HTML decks:

- open locally in a browser
- capture screenshots at desktop and narrow/mobile-ish widths if responsive
- check for blank renders, clipped text, overlap, and broken assets

For PPTX decks:

- verify the file opens or can be parsed
- inspect slide thumbnails or exported images when possible
- check that important text remains editable when editability was required
- confirm image paths are embedded and not linked to local-only temporary files

For all decks:

- check `outline.md` against the final deck: every planned slide is present or intentionally removed
- check style references were adapted consistently when provided
- check generated and sourced image assets are listed in `references/asset-manifest.json`, selected, and copied into workspace paths
- read slide thumbnails in order
- ensure each slide has one dominant message
- remove placeholder text and unused assets
- run a five-axis design review: thesis fit, visual hierarchy, craft quality, functional clarity, originality
- report any unverified assumptions or tool limitations

## Practical Patterns

### AI Plate Prompt Template

Use this shape and adapt it:

```text
Use $imagegen.
16:9 premium presentation background, [subject/metaphor], subtle [material/light],
clean negative space in [safe zone], abstract wireframe structure, no text, no logos,
low contrast behind content area, crisp but not busy, suitable for executive slide deck.
Text policy: no readable text inside the image; all text, numbers, labels, and claims are overlayed later in HTML/Canvas/PPTX.
```

### Precision Layer Rule

If a viewer may need to copy, edit, audit, or trust it, draw it with code or native PPTX objects.

If it only needs to set mood, context, texture, or visual metaphor, let the image model do it.

### File Layout

Use a clear deck workspace:

```text
deck-name/
  outline.md
  assets/
    plates/
    generated/
    logos/
    sources/
    style-references/
  references/
    style-reference-analysis.md
    visual-system.md
    imagegen-prompt-pack.md
    design-review.md
  html/
    index.html
  pptx/
    build.*
    output/
```

Keep generated plates separate from source assets so final cleanup is safe.
